LangGraph:
when a line becomes a map

LCEL chains run left to right, once. Real AI apps need to branch, loop, and remember. LangGraph models your app as a graph of small functions sharing one state — the engine real agents run on.

State + Nodes + Edges · runs on free Groq · the runtime behind Day 25's ReAct agent.
Where we hit a wall

Three things a | chain cannot do

Day 21's prompt | model | parser is a straight line: data flows one way, one time. But listen to what real apps ask for:

  • Branch — "if it's a refund question, send it to billing; if it's a crash, to tech support."
  • Loop — "if the answer isn't good enough, try again" (Day 23's tool loop was a hand-written while!).
  • Remember — "continue the conversation where we left off yesterday."
The tell

The moment your app description contains "if", "until", or "remember" — a straight chain is the wrong shape. You need a graph.

The one big idea

Your app = a flowchart that runs

LangGraph lets you draw your app as a flowchart in code: boxes (steps) and arrows (what runs next), all reading and writing one shared notebook (the state). Then it executes that flowchart for you.

Dashed arrows = a decision. Which path runs is decided by your data, while the app runs.

The whole framework in 3 words

State · Node · Edge

📓 State

One shared dictionary that flows through the graph. Every step reads it, every step can update it. class State(TypedDict)

🔧 Node

A plain Python function: state in → the keys it changed out. No magic, no classes to inherit. def step(state) -> dict

🔀 Edge

A wire: "after this node, run that one." Fixed (add_edge) or a runtime choice (add_conditional_edges).

Say it back

Nodes are functions. Edges are wires. State is a dict. Everything else today is detail.

Piece 1 · State

One notebook, passed hand to hand

from typing import TypedDict class State(TypedDict): message: str # what came in category: str # filled by classify reply: str # filled by a handler

The state is the contract between your nodes: it lists every field any step may read or write.

TypedDict (Day 8!) means it's still just a dict at runtime — but your editor and LangGraph both know its shape.

Mental picture

Imagine a case file at a support desk. Each employee (node) reads the file, writes their bit, and passes it on. Nobody carries private notes — everything lives in the file.

Piece 2 · Nodes

A node returns only what it changed

def classify(state: State) -> dict: text = state["message"].lower() # READ anything category = "billing" if "refund" in text else "general" return {"category": category} # WRITE only your keys

You do not return the whole state. Return a small dict of the keys you changed; LangGraph merges it into the state before the next node runs.

✅ Do

return {"category": "billing"} — small, honest, mergeable.

❌ Don't

Mutate state in place, or return unrelated keys. Updates should say exactly what changed.

Piece 3 · Edges — and the full recipe

Every graph you'll ever build, in 6 lines

from langgraph.graph import StateGraph, START, END builder = StateGraph(State) # 1. what shape is the state? builder.add_node("classify", classify) # 2. steps (name + function) builder.add_edge(START, "classify") # 3. wires builder.add_edge("classify", END) graph = builder.compile() # 4. freeze it into a runnable app graph.invoke({"message": "hi"}) # 5. run: dict in -> final state out
Familiar?

The compiled graph is a Runnable — the same .invoke() / .stream() interface as every LCEL chain from Day 21. A graph is a chain that learned new moves.

Move 1 · Branching

Conditional edges: the graph chooses

def route(state) -> str: return state["category"] builder.add_conditional_edges( "classify", # after this node route, # ...call this {"billing": "billing", "technical": "technical", "general": "general"})

A router is a tiny function that looks at the state and returns a name. The map turns that name into the next node.

  • Nodes change state; routers pick paths. Keep the jobs separate.
  • Today the router is an if. On Day 25, the model makes this choice — same wiring.
Move 2 · Loops

An arrow pointing backward = "try again"

builder.add_conditional_edges("check", keep_going, {"loop": "draft", # ← backward edge: the cycle "done": END})

Always give a loop a way out: an attempt counter in state + LangGraph's recursion_limit as the seatbelt.

Move 3 · Accumulating state

Reducers: append, don't overwrite

By default a node's update replaces a field. For lists — a chat history, an event log — you want updates to add on. A reducer annotation tells LangGraph how to merge.

Default — last write wins
class State(TypedDict): log: list # node returns {"log": ["b"]} # state["log"] == ["b"] 😬 lost "a"
With a reducer — it appends
from operator import add class State(TypedDict): log: Annotated[list, add] # node returns {"log": ["b"]} # state["log"] == ["a", "b"] ✅
For chat, it's prewritten

add_messages is the reducer for conversations, and MessagesState is a ready-made state with messages: Annotated[list, add_messages] already inside. Day 16's "append every turn" — now automatic.

Move 4 · Memory across calls

Checkpointer + thread_id = a bot that remembers

from langgraph.checkpoint.memory \ import MemorySaver graph = builder.compile( checkpointer=MemorySaver()) cfg = {"configurable": {"thread_id": "aditi"}} graph.invoke( {"messages": [("human", "Hi!")]}, cfg) # send ONLY the new msg
  • The checkpointer saves the state after every step.
  • The thread_id names the conversation. Same id → history reloads; new id → clean slate.
  • Each turn you send only the new message — the graph brings back the rest.
Scale it later

MemorySaver = RAM. Swap in a SQLite/Postgres saver and the same code survives restarts. That one line is your upgrade path.

Put side by side

Chain vs graph — when to reach for which

You need to…LCEL chainLangGraph
Run fixed steps in order (prompt → model → parser, RAG)✅ simplest✅ works
Branch on the data at runtime❌✅ conditional edges

Rule of thumb: straight line → chain. Any "if / until / remember" → graph.

Recap

Six things to walk away with

  • State is a shared TypedDict; nodes are functions returning only the keys they changed; edges are wires.
  • The recipe: StateGraph → add_node → add_edge → compile → invoke.
  • Branch with add_conditional_edges + a router that returns a name.
  • Loop with an edge pointing backward — and always an exit + attempt cap.
  • Make lists accumulate with reducers: Annotated[list, add], add_messages, MessagesState.
  • Remember with compile(checkpointer=…) + a thread_id per conversation.

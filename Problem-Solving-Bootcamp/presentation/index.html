Programming in general · 30 minutes

"I understand every solution.
Why can't I write one?"

The most common wall in learning to program — why it happens, why it's not a talent problem, and the exact practice routine that breaks it.

Spoiler: you've been training the wrong muscle.
The symptom

Sound familiar?

  • You read a problem… and stare at a blank file. Nothing comes.
  • You peek at the solution — and understand everything, instantly, in one go.
  • "So I clearly get it… why couldn't I produce it?" You conclude you're bad at this.

This is the classic wall — so common it has a nickname: tutorial hell. Almost everyone who learns to program hits it. Almost nobody is told why it happens.

The good news

Understanding solutions instantly means your comprehension is fine. The problem is precise, well-understood, and completely fixable — starting today.

The core issue

Reading code and writing code are two different muscles

👀 Reading = recognition

Every line arrives with its context. Your only job is to nod: "yes, that step makes sense." It's passive — like recognizing a face.see: for ch in text: ... think: "ah yes, a loop. obviously."

✍️ Writing = generation

You must invent the steps from nothing: break the problem down, recall the right tool for each step, put them in order.blank file. think: "...where do I even start?"

Watching 100 solutions gives you 100 reps of recognition and zero reps of generation. That's why the solution "makes sense in one go" while the blank file stays blank.

Why it fools everyone

The illusion of competence

"It makes sense" feels exactly like "I could have done it." Psychologists have a name for this trap — the illusion of competence. The feeling is real; the skill isn't there yet.

🏏 You already know this

Watching Kohli bat, every shot looks obvious — of course he played that ball there. Hand you the bat and… nothing transfers. Watching trains the eye, not the arm.

📖 Same trap, exams

Re-reading a chapter feels like studying — it all "makes sense." Then the exam asks you to produce it from a blank page, and it's gone. Recognition ≠ recall.

Code is no different: the solution file is someone else's innings. Understanding it proves you can follow — not that you can bat.

The trap, step by step

The tutorial-hell loop

Try a problem→ Stuck (feels bad)→ Peek at solution (feels productive!)→ "Ohh, makes sense!"→ Next problem: stuck again

The loop is self-reinforcing because the feelings lie to you: struggling feels like failure, and reading the solution feels like progress. So you bail out earlier each time.

But every early peek skips exactly one rep of the skill you're missing. A hundred peeks later: great at reading, still frozen at blank files.

What's actually missing

The solution hides the hard part

Solving = two jobs. The solution file only shows the second one:

Job 1 · Decomposition (invisible)
"count words appearing twice+" ↓ hmm… I need word counts ↓ a dict: word -> count ↓ then keep counts >= 2
Job 2 · The code (what you see)
counts = {} for w in words: counts[w] = counts.get(w, 0) + 1 repeats = [w for w, c in counts.items() if c >= 2]

When you read a solution, someone already did Job 1 for you — that's precisely why it reads so smoothly. Your gap is almost never Python syntax. It's decomposition: turning a fuzzy ask into steps.

The fix

Change what counts as a "rep"

❌ Not a rep

Lines of solution read. Videos watched. Tutorials followed along. All recognition — comfortable, and useless for generation.

✅ A rep

Minutes spent stuck, planning on paper, trying, failing, and trying again — before any solution. Uncomfortable, and it's the whole game.

Research on learning calls this productive struggle: students who wrestle with a problem first learn more from the very same solution than students shown it immediately. The struggle isn't the obstacle to learning — it is the learning.

Five rules make it concrete →

Rule 1 of 5

Plan before code — English first

No Python until the steps exist in plain English, written as comments in your file. (Days 1–5 students: this is the 5-step method — Understand → Example by hand → Plan → Code → Test.)

# STEP 1: split the text into words # STEP 2: count each word with a dict # STEP 3: keep the words with count >= 2 def repeated_words(text): ...
Why this works

If you can't write the plan, you've found your real gap — and it wasn't syntax. Now you know what to actually work on: trace an example by hand and watch what your own brain does. The plan is the program; the code just types it.

Rule 2 of 5

The 20-minute struggle timer

No hints, no solutions, no "just a quick look" before 20 minutes of genuine attempt. Planning on paper counts as attempting. Scrolling your phone doesn't.

  • Minute 3 — "I have no idea." Normal. Keep restating the problem.
  • Minute 8 — "Maybe if I…" The decomposition muscle is now actually working.
  • Minute 15 — a half-plan exists. This half-plan is worth more than ten read solutions.
Say it with us

Stuck is not a signal that you're bad at this. Stuck is the rep.

Rule 3 of 5

Climb the hint ladder — don't jump off the cliff

Going straight from "stuck" to "full solution" is an all-or-nothing cliff. A hint ladder unsticks you while leaving the decomposition to you. Read one rung, then go back and try:

🪜 Hint 1

Restates the problem + names the tool. "You need to track two values while looping."

🪜 Hint 2

The plan in pseudocode — the shape of the answer, no Python yet.

🪜 Hint 3

Only the trickiest 2–3 lines. You still assemble the rest.

Every problem in today's session ships with exactly this ladder (HINTS.md). The full solution exists too — as a last resort, not the next step.

Rule 4 of 5

Read a solution? Fine — now own it

Reading solutions isn't forbidden. Reading them and stopping there is. Two moves turn a read solution into a real rep:

🔄 Cover & rewrite

Close the solution. Blank file. Rewrite the whole thing from memory today. Where you stall is exactly what you hadn't actually absorbed.

🔀 Do the variant

Immediately solve a twist: solved second largest? Now do third. Built the encoder? Write the decoder. Can't? You recognized it — you didn't own it.

The honest test

"I understood the solution" is a feeling. "I rewrote it from blank and solved the variant" is evidence.

Rule 5 of 5

Trace before write

Writing code means doing two jobs at once: inventing steps and simulating whether they work. If you can't mentally run code, your plans have nothing to check against — of course the blank file feels impossible.

The daily 5-minute drill
count = 0 for ch in "banana": if ch == "a": count += 1 else: count = 0 print(count) # predict FIRST!
Why it compounds
  • Predict on paper → run → compare.
  • Wrong? You just found a precise gap in your mental model — free.
  • A fast, trusted simulator makes planning dramatically easier.

Beyond today

This is how you practice from now on — everywhere

  • Every exercise, every course, every language: plan first, struggle 20, climb hints, rewrite + variant.
  • One or two problems a day beats a weekend binge — generation grows by frequency, not volume.
  • Re-solve old problems from blank a week later. If it flows, it's yours now.
  • Watch out for AI autocomplete too — accepting generated code you couldn't have written is the same trap with better tooling. Use AI to review your plan, not to replace it.
